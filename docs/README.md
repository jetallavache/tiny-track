### Структура проекта

Здесь мы делим код на несколько подкаталогов, каждый из которых отвечает за отдельную часть логики:

- **`src/api/`** — HTTP API и WebSocket сервер.

- В этом каталоге будет код для обработки запросов, взаимодействия с WebSocket и REST API. Можно будет описать структуру роутов, обработчики для разных типов запросов.

- Пример файлов:

- `http_server.c` — HTTP-сервер (если нужен).

- `websocket.c` — сервер для WebSocket-соединений.

- `api_utils.c` — вспомогательные функции для работы с API.

- **`src/collector/`** — Сборщик метрик.

- Этот каталог будет содержать код для сбора информации из `/proc` и других системных источников.

- Пример файлов:

- `cpu_metrics.c` — сбор статистики по процессору.

- `mem_metrics.c` — сбор статистики по памяти.

- `net_metrics.c` — сбор статистики по сети.

- **`src/protocol/`** — Протокол бинарных пакетов.

- Код для сериализации/десериализации данных в бинарном формате.

- Пример файлов:

- `packet_format.h` — структуры заголовков пакетов.

- `packet_serializer.c` — функции сериализации/десериализации.

- `binary_protocol.c` — сам протокол передачи данных.

- **`src/server/`** — Основной сервер.

- Тут будет основной event loop, работа с сокетами через epoll, а также запуск всех подсистем (API, сбор метрик).

- Пример файлов:

- `server.c` — основной сервер, event loop, работа с epoll.

- `metrics_server.c` — сервер, который принимает данные от клиента.

- `utils.c` — вспомогательные функции (например, для работы с файловой системой, логирования и т.д.).

- **`src/utils/`** — Вспомогательные утилиты.

- Вспомогательные функции (например, парсинг конфигураций, логирование и т.д.).

- Пример файлов:

- `file_utils.c` — функции для работы с файлами (например, чтение /proc).

- `logging.c` — логирование.

- В `backend/utils/` можно добавить функции для логирования, например, с использованием `syslog`.

### Диаграмма последовательности (упрощённо)

```
Клиент                     Сервер
   |   connect (WS)            |
   | ------------------------> |
   |                           |
   |   <---- PKT_CONFIG -------|
   |                           |
   | -- PKT_CMD (interval=2s)->|
   |                           |
   |   <---- PKT_ACK ----------|
   |                           |
   |   <---- PKT_METRICS ------|
   |   <---- PKT_METRICS ------|
   |   <---- PKT_ALERT --------|
   |                           |
```


## Общий цикл общения сервер ↔ клиент

### 1. Подключение

- Клиент открывает WebSocket-соединение.

- Сервер добавляет подключение в свой список клиентов.

- (Опционально) Сервер сразу отправляет текущий конфиг (`PKT_CONFIG`), чтобы клиент знал, с какими параметрами работает мониторинг.


---

### 2. Синхронизация конфига

- **Сервер → клиент**  
Отправляется `PKT_CONFIG` с текущим интервалом и настройками.

- **Клиент → сервер**  
Может отправить `PKT_CMD` с изменением интервала или включением/выключением алертов.


Пример:

1. Сервер: `PKT_CONFIG { interval_ms=1000, alerts_enabled=1 }`

2. Клиент: `PKT_CMD { cmd_type=1, interval_ms=2000 }`

3. Сервер применяет новое значение и (опционально) подтверждает `PKT_ACK`.


---

### 3. Поток метрик

- Сервер раз в `interval_ms` собирает метрики из `/proc`.

- Сериализует в `PKT_METRICS`.

- Рассылает всем клиентам.


Пример:
`PKT_METRICS { cpu=23.5, mem=41.2, load1=0.85, ... }`

---

### 4. Алерты

- При срабатывании правила (например, CPU > 90%) сервер формирует пакет:
`PKT_ALERT { level=2, message="CPU > 90%" }`

- Отправляет всем клиентам или только тем, у кого включены алерты в конфиге.


---

### 5. Команды от клиента

Клиент может присылать `PKT_CMD`:

- `cmd_type=1` → новый интервал обновления

- `cmd_type=2` → включить/выключить алерты

- (в будущем) другие команды (например, запросить одноразовый снимок метрик).


Сервер:

1. Проверяет команду.

2. Применяет изменения.

3. Может отправить `PKT_ACK`, чтобы клиент понял, что команда применена.


---

### 6. ACK (подтверждения)

Опциональный механизм:

- После критичных действий (изменение интервала, включение алертов) сервер отправляет `PKT_ACK`.

- Payload может содержать код результата:

```c
struct ack_payload {
   uint8_t cmd_type;  // подтверждённая команда
   uint8_t status;// 0=ok, 1=error
};
```
